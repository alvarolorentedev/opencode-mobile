# Bellows API client and settings integration

The change adds a standalone HTTP client for a LiteLLM proxy ("Bellows") alongside the existing @opencode-ai/sdk, plus a settings UI section that exposes connection configuration, budget monitoring, and model listing. The client is a thin fetch wrapper with typed endpoints; the UI is a new `BellowsSection` card rendered between the existing Connection and AI Defaults sections. The @opencode-ai/sdk remains untouched as the primary backend.

Watch for: API key stored in plaintext in the same settings object that may be serialized to AsyncStorage (confirmed). No connection-state feedback beyond "Test Connection" loading spinner — silent failures leave the user guessing (confirmed). Bellows credentials live in `OpencodeConnectionSettings` rather than a dedicated settings slice, coupling an unrelated backend's auth to the opencode context (confirmed).

## High-level view

The Bellows client (`lib/bellows/client.ts`) is a zero-dependency fetch layer that mirrors LiteLLM's REST surface. It introduces its own `BellowsConnectionSettings` type but the actual values are stored inside `OpencodeConnectionSettings` as optional fields, creating a soft coupling between two unrelated backends.

The settings UI adds a single card combining connection configuration, budget progress display, and model list. State lives entirely in local `useState` hooks — budget/model data is fetched on-demand only and lost on re-mount, with no error feedback when requests fail.

Security posture is minimal: the API key is a plaintext field in the same settings blob that gets round-tripped through the provider context, likely serialized to device storage without encryption.

<summary>Issues (5)</summary>

1. **Plaintext API key in settings blob** — The bellowsApiKey sits in `OpencodeConnectionSettings` which is serialized by the existing provider persistence layer. Use SecureStore (expo-secure-store) or at minimum mark the field as sensitive so it isn't included in debug dumps.
2. **Silent fetch failures** — `refreshBellows` swallows errors via `.catch(() => null)` / `.catch(() => [])` with no user feedback. Surface a snackbar or error state so the user knows the connection failed and why.
3. **No connection test result display** — The "Test Connection" button shows a spinner while loading but provides no success/failure indication after the request completes. A brief status badge or snackbar would close the feedback loop.
4. **Credentials coupled to opencode settings type** — `bellowsServerUrl` and `bellowsApiKey` are optional fields on `OpencodeConnectionSettings`. This means any consumer of that type carries Bellows baggage. A separate settings slice (or at minimum a nested object) would keep the surface clean.
5. **Budget endpoint response shape assumption** — `bellowsBudgetInfo` expects `{ current_spend, max_budget }` directly from the response body. LiteLLM's actual `/budget/info` response nests these under a different key depending on version. If the shape doesn't match, the UI silently shows nothing (likely).

## Budget response shape fragility

`bellowsListModels` handles two response shapes (array or `{ data: [] }`) — a pragmatic defense against LiteLLM version differences. `bellowsBudgetInfo` makes no such accommodation: it casts the response directly to `BellowsBudgetInfo`. If LiteLLM wraps budget data in a container object (which some versions do), the client returns an object with `current_spend: undefined` and the UI renders `$NaN`. Adding a similar shape-normalization guard would prevent silent display corruption.

## Credential handling and storage

The API key is stored as a plain string field (`bellowsApiKey`) on `OpencodeConnectionSettings`, managed by the opencode provider context and almost certainly persisted to AsyncStorage — an unencrypted SQLite database on Android accessible to root or backup extraction.

For the default development key (`sk-anvil-safe-key`) this is tolerable, but the field is user-editable. The moment someone enters a real LiteLLM master key, it's stored unencrypted on device. Expo provides `expo-secure-store` for exactly this case: read/write the key via SecureStore and keep only a "key is set" flag in the settings blob.

The `secureTextEntry` prop on the input prevents shoulder-surfing but does not prevent serialization.

## Error handling in settings refresh flow

```typescript
const [budget, models] = await Promise.all([
  bellowsBudgetInfo(bellowsSettings).catch(() => null),
  bellowsListModels(bellowsSettings).catch(() => []),
]);
```

Both calls swallow errors completely. If the server is unreachable, returns 401, or returns malformed JSON, the user sees the spinner stop and nothing else. No error message, no status indicator, no differentiation between "server returned no budget" and "server is down." The `BellowsSection` component has no error prop and no way to display failure state.

The "Test Connection" button becomes meaningless without success/failure feedback — it's indistinguishable from "Refresh Data." A minimal fix: capture the error, expose it via state (`bellowsError: string | null`), and render it in the section. The Snackbar infrastructure already exists in the settings screen.

## Settings type coupling

Embedding `bellowsServerUrl` and `bellowsApiKey` directly on `OpencodeConnectionSettings` means every consumer of that type carries Bellows baggage. Today it's a minor aesthetic issue; it becomes a maintenance burden when Bellows needs more settings (timeout, retry policy, model alias maps) or when a second alternative backend is added. A nested `bellows?: { serverUrl: string; apiKey: string }` object or a parallel settings type would maintain separation.

## Progress bar type assertion

```typescript
width: `${spendPercent}%` as any,
```

React Native's `ViewStyle` type doesn't accept string percentages for `width` on all platforms. The `as any` escape works at runtime but bypasses the type system. Using `DimensionValue` from `react-native` (available since RN 0.71) would keep type safety: `width: \`${spendPercent}%\` as DimensionValue`.

## Model list rendering without virtualization

The model list renders all items in a `List.Section` inside a `ScrollView`. For deployments exposing 100+ model aliases, all `List.Item` components mount simultaneously. Not blocking for a first pass with typical 5-20 model counts, but worth noting if the list is expected to grow.
